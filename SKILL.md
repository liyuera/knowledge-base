# Obsidian 知识库管理 Skill

> 版本: 2.1.1 | 开源 | MIT

## 配置

本 skill 通过 `~/.claude/knowledge-base.json` 管理所有用户配置。运行 `install.sh` 自动完成首次配置。

### 配置优先级

环境变量 > 配置文件 > 默认值

| 配置项 | 环境变量 | 配置文件 key | 默认值 |
|--------|----------|-------------|--------|
| Vault 路径 | `OBSIDIAN_VAULT_PATH` | `vaultPath` | — (未配置时存储到项目 `.claude/knowledge-base/`) |
| KB 根目录 | `OBSIDIAN_KB_ROOT` | `kbRoot` | `50-工作日志` |
| 语言 | `OBSIDIAN_KB_LANGUAGE` | `language` | `zh` |

### 运行时路径解析规则

1. 先检查环境变量 `OBSIDIAN_VAULT_PATH`
2. 再读取配置文件 `~/.claude/knowledge-base.json` 的 `vaultPath` 字段
3. 若均未配置，fallback 到项目本地存储 `.claude/knowledge-base/`

### Vault 未配置时的行为

若 vault 路径无法解析：
- KB 文件写入到项目 `.claude/knowledge-base/<分支>/<YYYYMMDD>/` 目录
- 技术性问题记录到项目 `.claude/knowledge-base/technical/`
- 功能完整可用，仅存储位置不同

## Hooks 依赖

本 skill 需要以下 hooks 才能自动运行。`install.sh` 会自动将它们添加到 `~/.claude/settings.json`：

| Hook | 作用 |
|------|------|
| SessionStart | 注入 KB 内容、个人画像、初始化指令为系统上下文 |
| UserPromptSubmit | 收集用户消息到个人画像目录 + 注入 KB 日志格式提醒 |
| Stop | 调用 `check-kb.sh` 兜底校验今日 KB 文件状态 |

## 前置检查（阻断级，每次会话必须执行）

| # | 检查项 | 行为 |
|---|--------|------|
| 1 | 定位知识库路径 | 按配置优先级解析路径，不存在则自动创建 |
| 2 | 加载上下文 | 读取最近 5 个日期的 .md 文件 + `.claude/MEMORY.md` |
| 3 | 确认 MEMORY.md | 不存在则从 CLAUDE.md 和 git log 自动创建模板 |
| 4 | 扫描组件级规则 | 加载 `src/` 下所有 `CLAUDE.md` 和 `.context.md` |
| 5 | 提取待办项 | 列出所有 `[ ]` 未解决项，在回答时主动关注 |

## 目录结构

```
<Vault>/<KB_ROOT>/
├── <项目名>/<git分支>/<YYYYMMDD>/<YYYYMMDD>.md   # 项目知识库
├── 个人画像/
│   ├── 个人画像.md                                # 用户个人画像（全局）
│   └── <项目名>/<git分支>/<YYYYMMDD>/<YYYYMMDD>.md # 用户消息记录
└── <YYYYMMDD>.md                                  # 非项目级日志
```

## 内容分类记录规则

- **主动收集**: 在所有对话中识别用户提出的技术问题和业务问题，自动记录到对应位置
- **技术性问题**: 若 vault 已配置则记录到 `10-前端开发/技术学习/学习笔记/`，否则记录到项目 `.claude/knowledge-base/technical/`
- **项目或业务问题**: 记录到知识库目录，按项目-分支-日期归档
- **分类优先级**: "技术性问题" > "项目业务问题" > "可选记录"。禁止以"可选"为由跳过技术知识点的记录
- **纯咨询 / 解释性问题**: 仅当不涉及具体技术概念时可选记录。一旦涉及具体 API、语法、框架特性，一律视为技术性问题

## 汇报前写入规则（阻断级，优先级最高）

**代码修改完成后，必须先写入知识库，再向用户汇报"完成"。两者的执行顺序不可颠倒。禁止先汇报再补写。**

### 执行流程

```
代码修改完成 → 判断是否需要写入 → 写入知识库文件 → 更新 MEMORY.md（如需要） → 汇报用户
```

### 判断标准

一个功能、一个修复、一次重构的所有文件改完后，立即写入知识库。触发时机：
- 用户切换话题
- 用户开启新任务
- 用户询问"还有多少"/"进度"且当前任务已完成
- 用户明确表示当前任务完成

### 自检规则

**每次准备汇报"完成"之前，必须执行以下自检：**
1. 我是否已经创建/更新了今天日期的知识库文件？
2. 如果没有，我现在是否应该跳过？理由是什么？
3. 如果理由不是"纯格式/位置/文案变更"，必须立即写入

### 日期获取规则（阻断级）

**禁止凭记忆或推测日期。** 任何涉及日期的操作（创建目录、写入文件、KB 日志中的日期引用），必须先执行 `date +%Y%m%d` 获取系统日期。

### KB 日志校验规则（阻断级）

KB 日志中的 `✅ 已写入` **必须先验证再输出**。执行流程：

1. 写入知识库文件
2. **立即执行验证命令**：`grep -c "### #N" <今日KB文件路径>` 确认条目数增加
3. 验证通过后才能输出 `📋 KB | ✅ 已写入 | ...`
4. 验证未通过 → 输出 `📋 KB | ❌ 写入失败 | <原因>` → 重试写入

**禁止行为**：
- 禁止写入后不验证直接声称"已写入"
- 禁止凭记忆判断写入状态
- 禁止用 `✅ 已写入` 作为意图表达

### 违反后果

若出现以下情况，视为该次执行失败，用户反馈优先采纳：
- KB 日志声称"已写入"但文件实际无内容
- 日期错误（未用 `date +%Y%m%d` 校验）
- 先汇报再补写

## 回滚代码写入规则（阻断级，优先级最高）

**代码回滚是有意义的变更，必须写入知识库。禁止以"撤销后代码无变化"为由跳过。**

### 判断标准

满足以下任一条件视为回滚，必须写入：
- 用户明确要求"回滚"/"撤销"/"还原"/"revert"某次修改
- `git checkout` / `git restore` / `git revert` / `git reset` 操作
- 手动将文件内容恢复到修改前的状态
- 删除之前新增的文件，恢复之前删除的代码

### 执行流程

```
检测到回滚操作
  → 定位被回滚的原 KB 条目（搜索今日 + 近 3 天文件）
  → 在原条目末尾追加标记行：`> ⚠️ 已于 YYYY-MM-DD 回滚，以下条目描述的内容不再有效`
  → 写入新的 [回滚] 条目，引用原条目编号
  → 汇报用户
```

### 回滚条目格式

```markdown
### #N [回滚] 回滚 <原条目概述>
- **操作**: 撤销了 #X [类型] <原条目标题>，代码恢复至修改前状态
- **文件**: 回滚涉及的文件列表
- **原因**: <用户反馈的问题 / 方案不可行 / 引入新问题>
- **结果**: 已标记 #X 条目为已回滚，代码恢复

---
```

### 无法定位原条目时

若无法找到被回滚的原 KB 条目（如原条目在更早日期），仍必须写入 `[回滚]` 条目，原条目标记由后续会话补做。

### 自检追加项

准备汇报"完成"之前，追加自检：
4. 本次是否有回滚操作？若有，原 KB 条目是否已标记回滚？

## 跳过规则

以下情况不写入知识库：
- 纯位置调整：移动代码块 / 列顺序 / import 排序，逻辑无变化
- 纯格式变更：缩进、换行、注释增删
- 单行文字修改：标签文案、提示语微调

## 写入格式（原子条目）

每条记录为固定格式的独立条目：

```markdown
### #N [类型] 一句话概述
- **操作**: 具体做了什么（一句话）
- **文件**: `path/to/file.js:行号`
- **原因**: 为什么这样做（可选）
- **结果**: 解决了什么 / 引入了什么

---
```

类型枚举：`[修复]` `[新增]` `[重构]` `[决策]` `[回滚]`

## Edit 缓存刷新规则（阻断级）

**每次对 KB 文件执行 Edit 后，该文件的 Read 缓存不会自动失效。** 下一次 Read（包括带 offset/limit 的 Read）返回的是首次读取的缓存内容，而非磁盘实际内容。old_string 基于过时缓存必然匹配失败，Edit 静默跳过但不会报错。

### 多条目写入流程

当需要对同一 KB 文件执行多次 Edit 时：

1. **首次写入前**：Read(full) 全文件，获取初始状态
2. **Edit 写入条目** — 追加第一个 `### #N` 条目
3. **Read(full) 全文件** — 强制刷新缓存，获取包含新条目的完整内容
4. **逐条验证**：`grep "### #N" <今日KB文件路径>` — 确认本条已落盘，未找到则立即报错
5. **Edit 写入下一条目** — 基于刷新后的内容追加第二个条目
6. **重复步骤 3-5** — 每写入一个条目后：Read(full) → grep 验证 → Edit 下一条
7. **最终验证**：`grep -c "### #N" <今日KB文件路径>` 确认条目总数正确

### 失败检测与用户通知（阻断级）

Edit 失败时工具不会报错，因此**每一步必须主动检测，失败必须让用户看到**：

- 每次 grep 验证发现条目未落盘 → **立即在回复中输出** `📋 KB | ❌ 写入失败 | 条目 #N 未在文件中找到`，不得继续后续写入
- 最终条目数与预期不符 → 输出 `📋 KB | ❌ 写入失败 | 预期 N 条，实际 M 条`
- **禁止**检测到失败后不通知用户、静默重试或假装成功
- 失败通知必须在回复正文中出现，不能只在最后的 KB 日志行

### 禁止行为

- 禁止连续多次 Edit 同一个 KB 文件而不穿插 Read(full)
- 禁止使用 Read(offset, limit) 来"读取刚才写入的部分" — 缓存命中后 offset 指向旧文件内容
- 禁止信任 "file unchanged since last Read" 提示 — 那是缓存状态，不是磁盘状态
- 禁止 Edit 后跳过 grep 验证直接进入下一次 Edit

## MEMORY.md 同步规则

- 每次写入知识库条目后立即判断：是否引入了新约定、模式变更、新的已知问题、新的关键文件关联？若是，立即同步 MEMORY.md
- 常规 `[修复]` / `[新增]` 条目通常无需同步 MEMORY.md，仅 `[决策]` 或改变了项目约定的条目才同步
- 若新增内容不属于已有主题，在 MEMORY.md 中新建主题段落
- 每次 `git commit` 后，将提交日期、摘要、变更文件追加到 MEMORY.md 变更日志表格

## 过时知识标记规则

- 已修复的 `[ ]` 待办项改为 `[x]`，保留修复日期
- 若发现 MEMORY.md 或知识库中的事实与当前代码不一致，以代码为准并立即更新记录
- 回滚后必须在被回滚的原条目末尾追加：`> ⚠️ 已于 YYYY-MM-DD 回滚，以下条目描述的内容不再有效`
- 若原条目在更早日期的 KB 文件中，需跨文件追加回滚标记（Edit 该日期文件）

## 修改前自动回顾规则

- 修改任何文件前，先检查 MEMORY.md 中该文件是否属于某个"关联文件"列表
- 若属于，先读取 MEMORY.md 中对应的主题段落和注意事项，再动手修改
- 若目录下存在 `.context.md`，读取该文件获取目录级别的约束和已知问题

## 已初始化项目升级规则

对于已有 CLAUDE.md 但缺少知识库组件的项目，前置检查时自动补齐：
- `.claude/MEMORY.md` 不存在 → 从 CLAUDE.md 和 git log 提取已有信息填充
- 知识库目录不存在 → 自动创建，从 git log 近 5 个提交提取摘要写入
- 组件级 `CLAUDE.md` → 不自动创建

## KB 运行日志规则（阻断级）

**每次回答结束前最后一步**，必须输出知识库运行日志。无论回答内容是否涉及代码修改，一律输出。

格式：
```
📋 KB | <操作> | <详情>
```

操作枚举：
- `✅ 已写入` — 本次逻辑闭环已写入知识库（列出条目编号和概述）
- `⏭️ 无需写入` — 本次无代码变更、纯咨询、或命中跳过规则
- `⏭️ 已跳过` — 有代码变更但命中跳过规则（纯位置/格式/文案变更），注明原因

日志必须是回复的最后一行内容。

## 访问边界

- 仅读写当前项目知识库路径下的文件
- 禁止读取或写入其他项目的知识库文件
- 禁止读取或写入 vault 中非知识库目录的文件

## 个人画像功能 (v2.1.0)

> 通过收集每次对话中用户发送的消息，分析用户习惯、偏好、性格，生成个人画像文档，在每次会话中注入为系统上下文以提高 AI 输出精度。

### 消息收集机制

- **触发**: UserPromptSubmit hook 调用 `collect-user-message.sh`，每次用户发送消息时自动写入
- **存储路径**: `<Vault>/<KB_ROOT>/个人画像/<项目名>/<git分支>/<YYYYMMDD>/<YYYYMMDD>.md`
- **格式**: 按时间戳分段记录原始消息内容

### 个人画像生成与更新

**触发条件**（SessionStart 时注入指令）：
1. `个人画像/个人画像.md` 不存在 → 生成指令，要求 Claude 在本次会话中创建
2. 存在但距上次更新超过 7 天 → 刷新指令，要求 Claude 融入新消息
3. 存在且未过期 → 直接注入画像内容

**生成流程**：
1. 读取 `个人画像/` 下所有项目的消息记录（优先最近 30 天）
2. 按画像模板维度逐一分析
3. 生成/更新 `个人画像/个人画像.md`

```markdown
# 用户个人画像

- **技术深度**: （表层使用 / 深入原理 / 架构设计）

## 工作习惯
- **活跃时段**: 
- **项目类型**: 
- **代码风格**: 
- **开发流程**:

- **语言偏好**: （中文 / 英文 / 混合）
- **表达方式**: （简洁直接 / 详细解释 / 先结论后细节）
- **反馈模式**: （直接纠正 / 引导式提问 / 渐进式确认）
```

### 画像注入

`inject-kb-context.sh` 在 SessionStart 时读取 `个人画像/个人画像.md`，与 KB 上下文一起注入到系统上下文窗口。参考画像指导所有回答的语气、技术深度、解释粒度。
